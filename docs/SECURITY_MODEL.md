## Threat Model

KeePassEx is designed to protect against:

| Threat                                   | Protection                                                    |
| ---------------------------------------- | ------------------------------------------------------------- |
| Attacker with vault file but no password | Argon2id KDF + ChaCha20-Poly1305 encryption                   |
| Attacker with vault file + weak password | Memory-hard KDF (64 MB RAM, 2 iterations)                     |
| Attacker with vault file + hardware key  | Requires physical possession of hardware key                  |
| Malicious sync server                    | Vault encrypted before upload; server sees only ciphertext    |
| Clipboard sniffing                       | Auto-clear after 10 seconds (configurable)                    |
| Screen capture                           | Platform-level screen capture protection on mobile            |
| Memory dump                              | `ZeroizeOnDrop` on all sensitive types; mlock on key material |
| Tampered vault file                      | HMAC-SHA256 per-block integrity check                         |
| Brute force                              | Argon2id with configurable memory/time cost                   |
| Duress (forced disclosure)               | Decoy vault with separate password                            |
| Unauthorized plugin access               | WASM sandbox with explicit permission model                   |
| Browser extension leaking passwords      | Native messaging only; no passwords stored in extension       |

KeePassEx does **not** protect against:

- Attacker with full control of the device (OS-level compromise)
- Keyloggers capturing the master password at entry time
- Shoulder surfing
- Legal compulsion (no plausible deniability beyond decoy vault)

## Cryptographic Design

### Algorithm Selection

| Component            | Algorithm                      | Rationale                                     |
| -------------------- | ------------------------------ | --------------------------------------------- |
| Key derivation       | **Argon2id**                   | Memory-hard, GPU-resistant, OWASP recommended |
| Symmetric encryption | **ChaCha20-Poly1305**          | AEAD, constant-time, no padding oracles       |
| Block integrity      | **HMAC-SHA256**                | Per-block authentication, tamper detection    |
| Inner stream         | **ChaCha20**                   | Protected field encryption within XML         |
| Hash (HIBP)          | **SHA-1** (k-anonymity prefix) | HIBP API requirement; no security implication |
| Hash (internal)      | **SHA-256**                    | Key material hashing                          |
| Random               | **OS CSPRNG**                  | `getrandom` crate, platform-native            |

All algorithms are implemented via audited Rust crates:

- `argon2` — Argon2id KDF
- `chacha20poly1305` — ChaCha20-Poly1305 AEAD
- `sha2` — SHA-256
- `hmac` — HMAC-SHA256
- `rand` — CSPRNG

## Key Derivation

### Argon2id Parameters (defaults)

```
Memory:      65536 KB (64 MB)
Iterations:  2
Parallelism: 1
Output:      32 bytes
```

These parameters are stored in the KDBX header and can be increased for higher security at the cost of unlock speed.

### Key Derivation Flow

```
Master Password
      │
      ▼
  UTF-8 encode
      │
      ▼
  Argon2id(password, salt, memory=64MB, iterations=2)
      │
      ▼
  32-byte derived key
      │
      ├── Key File (XOR/hash combined if present)
      │
      └── Hardware Key response (HMAC-SHA1, if configured)
            │
            ▼
        Composite Key (SHA-256 of all components)
            │
            ▼
        Master Key (used to decrypt vault)
```

### Key File

A key file provides a second factor that is **not** derived from a password. It can be:

- A 32-byte random file (generated by KeePassEx)
- Any file (its SHA-256 hash is used)
- An XML key file (KeePass format)

The key file is combined with the password-derived key before encryption. An attacker needs **both** the vault file and the key file.

## Vault Encryption

### KDBX 4.x Format

```
┌─────────────────────────────────────────────────────┐
│ KDBX Header (plaintext)                             │
│   - Signature (magic bytes)                         │
│   - Version (4.x)                                   │
│   - Cipher ID (ChaCha20-Poly1305)                   │
│   - KDF parameters (Argon2id params + salt)         │
│   - Master seed (32 random bytes)                   │
│   - Encryption IV (12 bytes)                        │
├─────────────────────────────────────────────────────┤
│ Header HMAC (32 bytes)                              │
│   Authenticates the header against tampering        │
├─────────────────────────────────────────────────────┤
│ Encrypted Payload (ChaCha20-Poly1305)               │
│   ┌─────────────────────────────────────────────┐   │
│   │ Block 0: HMAC + data                        │   │
│   │ Block 1: HMAC + data                        │   │
│   │ ...                                         │   │
│   │ Block N: HMAC + data                        │   │
│   └─────────────────────────────────────────────┘   │
│   Each block has its own HMAC-SHA256 for integrity  │
└─────────────────────────────────────────────────────┘
```

### Decryption Process

1. Parse header (plaintext)
2. Derive master key from password + key file + hardware key
3. Verify header HMAC — **abort if invalid** (tamper detection)
4. Decrypt payload with ChaCha20-Poly1305
5. Verify each block's HMAC — **abort if any block is invalid**
6. Decompress (gzip) and parse XML

### Protected Fields

Passwords and other sensitive fields are additionally encrypted with a **ChaCha20 inner stream** within the XML. This means even if the XML is somehow extracted, passwords remain encrypted.

## Composite Key

The composite key combines multiple authentication factors:

```rust
pub struct CompositeKey {
    password_key: Option<Vec<u8>>,    // Argon2id(password)
    key_file_key: Option<Vec<u8>>,    // SHA-256(key_file_data)
    hardware_key: Option<Vec<u8>>,    // HMAC-SHA1(challenge, yubikey)
}

impl CompositeKey {
    pub fn build(&self) -> Result<Vec<u8>> {
        // SHA-256 of all present components concatenated
        let mut hasher = Sha256::new();
        if let Some(k) = &self.password_key { hasher.update(k); }
        if let Some(k) = &self.key_file_key { hasher.update(k); }
        if let Some(k) = &self.hardware_key { hasher.update(k); }
        Ok(hasher.finalize().to_vec())
    }
}
```

All components are required to be present at unlock time. Removing any component makes the vault inaccessible.

## Memory Protection

### ZeroizeOnDrop

All sensitive types implement `ZeroizeOnDrop` from the `zeroize` crate:

```rust
#[derive(ZeroizeOnDrop)]
pub struct ProtectedString(String);

#[derive(ZeroizeOnDrop)]
pub struct MasterKey([u8; 32]);
```

When these values are dropped (go out of scope), their memory is overwritten with zeros before deallocation. This prevents sensitive data from lingering in memory after use.

### Memory Locking

On supported platforms, key material is locked in memory using `mlock()` to prevent it from being swapped to disk:

```rust
// Prevents the OS from swapping key material to disk
libc::mlock(key_ptr, key_len);
```

### Guard Pages

The vault's in-memory representation uses guard pages on supported platforms to detect buffer overflows that could expose key material.

### Vault Locking

When the vault is locked (idle timeout, minimize, screen lock):

1. The master key is zeroed from memory
2. The decrypted vault data is cleared
3. Only the encrypted vault file path is retained

Re-opening requires re-entering the master password (or biometric unlock, which retrieves the key from the OS keychain).

## Clipboard Security

### Auto-Clear

Copied passwords are automatically cleared from the clipboard after a configurable delay (default: **10 seconds**).

```typescript
// After copying:
setTimeout(() => {
  invoke('copy_to_clipboard', { text: '', clearAfterSeconds: 0 });
}, clipboardClearSeconds * 1000);
```

### Clear on Lock

When the vault is locked, the clipboard is immediately cleared if it contains vault data.

### No Clipboard on Watch

Watch apps (watchOS, WearOS) do not copy passwords to the clipboard. Instead, they request the phone to copy via WatchConnectivity/DataClient, keeping sensitive data off the watch.

## Sync Security

### Encryption Before Upload

The vault is **always encrypted before leaving the device**. The sync provider (Google Drive, Dropbox, S3, etc.) only ever receives the encrypted `.kdbx` file.

```
Device                    Sync Provider
  │                            │
  │  vault.kdbx (encrypted)    │
  │ ─────────────────────────► │
  │                            │
  │  vault.kdbx (encrypted)    │
  │ ◄───────────────────────── │
  │                            │
  │  Decrypt locally           │
```

The sync provider **cannot** read vault contents. Even if the provider is compromised, the attacker only gets the encrypted file.

### Conflict Resolution

When both local and remote vaults have been modified:

1. **Merge** (default): CRDT-inspired merge keeps the newest modification per entry
2. **Keep Local**: Discard remote changes
3. **Keep Remote**: Discard local changes
4. **Ask User**: Present a diff and let the user decide

The merge algorithm never loses data — conflicting entries are kept with timestamps.

### Transport Security

All sync providers use HTTPS/TLS. The S3 provider uses AWS Signature V4 for request authentication.

## Browser Extension Security

### Native Messaging

The browser extension communicates with the desktop app via **native messaging** — a secure IPC channel provided by the browser. No passwords are stored in the extension itself.

```
Browser Extension          Desktop App (KeePassEx)
      │                           │
      │  getCredentialsForUrl     │
      │ ─────────────────────────►│
      │                           │  (vault is unlocked)
      │  { uuid, title, username }│
      │ ◄─────────────────────────│
      │                           │
      │  autofill(uuid)           │
      │ ─────────────────────────►│
      │                           │  (retrieve password)
      │  { username, password }   │
      │ ◄─────────────────────────│
```

**Passwords are only sent to the extension at fill time**, not stored in extension memory.

### URL Matching

The extension uses smart URL matching to prevent credential phishing:

- `github.com` matches `github.com` (exact domain)
- `github.com` matches `api.github.com` (subdomain)
- `github.com` does **not** match `github.com.evil.com`

### Content Script Isolation

The content script runs in an isolated world and cannot access page JavaScript. It only interacts with DOM elements to fill credentials.

## Emergency Access Security

### End-to-End Encryption

Emergency access uses asymmetric encryption so KeePassEx (the service) never sees vault contents:

1. **Grantor** generates an asymmetric key pair
2. **Grantee** provides their public key
3. Grantor encrypts vault key with grantee's public key
4. During emergency, grantee decrypts with their private key

The waiting period (default 7 days) allows the grantor to deny access if the request is unauthorized.

### Access Levels

- **View only**: Grantee can read entries but not modify the vault
- **Full takeover**: Grantee receives the vault key and can re-encrypt with their own password

## Plugin Sandbox

Plugins run in a **WebAssembly (WASM) sandbox** with explicit permissions:

| Permission            | What it allows                                     |
| --------------------- | -------------------------------------------------- |
| `read_entry_metadata` | Read entry titles, usernames, URLs (not passwords) |
| `read_passwords`      | Read passwords (requires explicit user grant)      |
| `write_entries`       | Create/modify entries                              |
| `network`             | Make HTTP requests                                 |
| `file_system`         | Read/write files                                   |
| `clipboard`           | Write to clipboard                                 |
| `notifications`       | Show notifications                                 |

Plugins cannot:

- Access the master key or encryption keys
- Communicate with other plugins
- Escape the WASM sandbox
- Access the OS directly

All plugin operations are logged in the audit log.

---

## Audit Log

The audit log records security-relevant events:

- Vault open/lock/save
- Entry view, create, modify, delete
- Password copy, OTP generation
- Emergency access requests/grants
- Sync operations
- Master password changes
- Hardware key add/remove
- Biometric unlock
- Failed unlock attempts

The audit log is stored **encrypted within the vault file** and is only accessible when the vault is unlocked. It uses a ring buffer (default: 1000 events) to prevent unbounded growth.

Failed unlock attempts are tracked to detect brute-force attacks. After 5 failed attempts, a configurable delay is imposed.

## Decoy Vault

The decoy vault provides protection under duress (e.g., being forced to reveal your password):

1. Create a separate `.kdbx` file with a different password
2. Populate it with realistic-looking fake entries
3. If forced to reveal your password, provide the decoy password instead

**Security properties:**

- The decoy vault is a standard KDBX file — indistinguishable from the real vault
- No metadata links the two vaults
- Accessing the decoy vault does not trigger any alert
- The real vault remains protected

**Limitations:**

- Does not protect against an attacker who knows you use KeePassEx and has multiple vault files
- Does not provide legal protection in jurisdictions with compelled decryption laws

## Platform-Specific Security

### iOS

- **Keychain**: Master key stored in iOS Keychain with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`
- **Screen capture protection**: `UIScreen.main.isCaptured` check + `UITextField.isSecureTextEntry`
- **App backgrounding**: Vault locked when app enters background
- **Face ID / Touch ID**: LAContext with `deviceOwnerAuthenticationWithBiometrics`
- **AutoFill**: Credentials never stored in AutoFill provider; fetched from vault at fill time

### Android

- **Keystore**: Master key stored in Android Keystore (hardware-backed on supported devices)
- **Screen capture protection**: `WindowManager.LayoutParams.FLAG_SECURE`
- **App backgrounding**: Vault locked on `onPause()`
- **Biometric**: `BiometricPrompt` API with `BIOMETRIC_STRONG` requirement
- **AutoFill**: `AutofillService` with `FLAG_SECURE` on sensitive views

### Desktop (Windows/macOS/Linux)

- **OS Keychain**: Master key stored in OS keychain (Windows Credential Manager, macOS Keychain, libsecret on Linux)
- **Memory locking**: `mlock()` on key material
- **Screen lock detection**: Lock vault on system screen lock event
- **Auto-type obfuscation**: Two-channel auto-type to defeat keyloggers

### watchOS / WearOS

- **No local storage**: Watch apps do not store vault data locally
- **WatchConnectivity / DataClient**: All data fetched from phone on demand
- **No clipboard**: Passwords are not copied to watch clipboard

## Known Limitations

1. **Master password at entry**: The master password is briefly in memory as a UTF-8 string before being processed by Argon2id. A memory dump at this exact moment could expose it.

2. **Auto-type**: Auto-type sends keystrokes to the active window. A malicious application could intercept these keystrokes.

3. **Clipboard**: Even with auto-clear, a clipboard manager or other application may have captured the password before it was cleared.

4. **Screen recording**: Screen capture protection prevents screenshots but may not prevent all screen recording software.

5. **Swap/hibernation**: On systems without full-disk encryption, key material in memory could be written to swap or hibernation files. Use full-disk encryption (BitLocker, FileVault, LUKS) to mitigate.

6. **Browser extension**: The browser extension requires the desktop app to be running and unlocked. If the desktop app is compromised, the extension is also compromised.

## Post-Quantum Cryptography

KeePassEx supports **hybrid post-quantum encryption** combining classical X25519 with CRYSTALS-Kyber-768 (NIST FIPS 203 / ML-KEM).

### Why Hybrid?

- If quantum computers break X25519 → Kyber-768 still protects
- If Kyber-768 has an undiscovered flaw → X25519 still protects
- Backward compatible: classical-only vaults can be migrated

### Implementation

```
Master Key
    │
    ▼
derive_pqc_keypair(master_key, HybridKyber768)
    → (x25519_keypair, kyber768_keypair)
    │
    ▼
encapsulate(public_keys)
    → x25519_shared_secret (32 bytes)
    → kyber768_shared_secret (32 bytes)
    → encapsulation (stored in KDBX header field 0x80)
    │
    ▼
HKDF-SHA256(x25519_shared || kyber768_shared)
    → combined_key (32 bytes)
    │
    ▼
ChaCha20-Poly1305 encrypt with combined_key
```

**Performance**: ~50ms overhead per vault open/save (negligible for interactive use).

**Compatibility**: PQC vaults can only be opened by KeePassEx. KeePass/KeePassXC will not open them.

## Zero-Knowledge Password Verification (ZKPV)

ZKPV allows fast password pre-check without running the full Argon2id KDF:

```
Setup (once):
  commitment = BLAKE3(password || random_salt)
  Store commitment + salt in vault header

Verification:
  candidate_commitment = BLAKE3(candidate_password || stored_salt)
  if candidate_commitment != stored_commitment → reject immediately
  else → proceed with full Argon2id KDF
```

**Benefits**:

- Instant rejection of wrong passwords (no 2-second Argon2id wait)
- No security reduction: commitment reveals nothing about the password
- Optional: can be disabled for maximum security

**Windows Credential Provider**: Uses ZKPV to verify the master password at the Windows login screen without running full Argon2id, keeping login fast.

## Steganography Security

When embedding a vault in a carrier file (PNG/JPEG/MP4/AVI):

1. Vault bytes are encrypted with ChaCha20-Poly1305 using a **separate steganography password**
2. Encrypted bytes are embedded using LSB (PNG) or metadata (JPEG/MP4/AVI)
3. A magic header `KPX\x00STG\x01` identifies embedded vaults

**Security properties**:

- The carrier file remains fully functional and visually identical
- Without the steganography password, the vault cannot be extracted
- The vault itself is still encrypted with the master password (double encryption)
- No metadata links the carrier file to KeePassEx

**Limitations**:

- Statistical analysis may detect LSB embedding in PNG files
- JPEG embedding is limited to 64KB (EXIF metadata)
- Not a substitute for full-disk encryption

## Vault Key Sharding (Shamir's Secret Sharing)

Vault key sharding splits the master key into N shards where any M shards can reconstruct it:

```
Master Key (32 bytes)
    │
    ▼
split_secret(key, threshold=M, total=N)
    → shard_1, shard_2, ..., shard_N
    (each shard is useless alone)
    │
    ▼
Distribute shards to N trusted parties/locations

Recovery:
    Any M shards → combine_shards([s_i, s_j, ..., s_k])
    → Master Key
```

**Implementation**: GF(256) finite field arithmetic with Lagrange interpolation (same math as AES S-box).

**Security properties**:

- Any M-1 shards reveal zero information about the master key
- Losing more than N-M shards permanently locks the vault
- Each shard is a standard file (`.kpxshard`) with no identifying metadata

**Use cases**:

- Corporate vault: require 3-of-5 executives to unlock
- Family vault: require 2-of-3 family members
- Backup: 5 shards in 5 different locations, any 3 can recover
